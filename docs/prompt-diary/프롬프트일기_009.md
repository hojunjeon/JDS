# [프롬프트 일기 #9] UI 리뉴얼 계획을 로드맵과 실행 계획으로 나누기

> JDS를 만들면서 AI와 함께 일하는 방식을 기록한다.
> 어떤 프롬프트가 들어왔고, 무엇이 바뀌었고, 그래서 어떤 결정을 남겼는지 적는다.

---

## 그 시점의 게임

UI 리뉴얼은 Phase 1 메뉴 흐름까지 구현되어 있었다. 하지만 루트 계획 문서와 상세 계획 문서의 상태 표시가 서로 다르게 읽혔다. `PLANS.md`는 완료처럼 보였고, 실제 `the former UI Renewal roadmap`는 Phase 1만 완료된 상태였다.

---

## 그날 상황

사용자는 이 문제가 단순한 체크박스 실수가 아니라, 계획 문서의 역할이 섞여 생긴 문제라는 점을 짚었다. 그리고 앞으로는 큰 작업을 `superpowers:writing-plans`로 나눠서 task별 체크리스트가 있는 실행 계획으로 관리하자는 방향을 제안했다.

---

## 입력된 프롬프트

> "페이즈별로 계획파일은 한번에 만들고 나눠서 구현할 수 있지?"

> "좋아 이 방식대로 진행해줘"

---

## 무엇이 변화했나

UI 리뉴얼 문서의 역할을 셋으로 나누기로 했다.

- `PLANS.md`: 전체 진행 상태를 보는 루트 인덱스
- `the former UI Renewal roadmap`: UI 리뉴얼 전체 로드맵과 Phase 상태표
- `docs/superpowers/plans/`: 실제 구현용 체크박스 실행 계획

Phase 2, Phase 3, Phase 4는 한 번에 계획 파일을 만들되, 구현은 Phase별로 끊어서 진행하기로 했다.

---

## 그래서 어떤 결정이 남았나

앞으로 JDS의 큰 작업은 다음 흐름을 기본으로 삼는다.

```text
brainstorming으로 방향 정하기
-> writing-plans로 실행 계획 쪼개기
-> Phase나 task 단위로 구현
-> 검증 후 다음 단계 진행
```

루트 계획의 `[x]`는 전체 작업이 끝났을 때만 쓴다. 일부 Phase가 끝난 것은 상세 로드맵이나 phase plan 안에서만 표시한다.

---

## 다음에 보고 싶은 것

실제 Phase 2 구현을 시작할 때 계획 파일의 체크박스가 잘 작동하는지 봐야 한다. 계획이 너무 길거나 실제 코드와 어긋나면 다음 Phase 계획부터 더 작고 단순하게 조정하면 된다.
