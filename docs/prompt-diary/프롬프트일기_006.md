# [프롬프트 일기 #6] 루트 PLANS.md로 진행 작업 인덱스 만들기

> JDS를 만들면서 AI와 어떻게 작업하는지 기록한다.
> 어떤 프롬프트가 들어왔고, 무엇이 바뀌었고, 그래서 어떤 결정이 생겼는지 적는다.

---

## 그 시점의 게임

UI Renewal은 Phase 1: Core Menu Flow 진행 중이었다. 세부 진행 상태는 `the former UI Renewal roadmap`의 체크리스트로 통합했지만, 세션 초기화 이후 어떤 계획 파일을 먼저 봐야 하는지는 루트에서 바로 드러나지 않았다.

---

## 그날 상황

사용자는 세션 초기화나 개발 환경 변경 이후 "진행상황 확인해줘"라고 말했을 때, Codex가 루트에서 활성 계획을 바로 찾고 세부 계획으로 들어갈 수 있는 구조를 제안했다.

---

## 입력된 프롬프트

> "루트 디렉토리에 PLANS.md 파일을 만들고 내용에는 계획 파일의 위치를 체크 박스로 만들어두는거야. [] 진행전 , [~] 진행 중, [x] 완료 이렇게."

---

## 무엇이 변화했나

루트에 `PLANS.md`를 추가했다. 이 파일은 활성 계획의 상위 인덱스이며, 상태 표기 `[ ]`, `[~]`, `[x]`의 의미를 명시한다. 현재 UI Renewal Phase 1은 `[~]` 진행 중으로 등록했고, 세부 계획 파일 위치와 다음 작업을 적었다.

`the former UI Renewal roadmap`에는 루트 계획 인덱스가 `PLANS.md`라는 사실과 두 파일의 상태를 동기화해야 한다는 문장을 추가했다.

---

## 그래서 어떤 결정이 있었나

앞으로 세션이 초기화되면 먼저 루트 `PLANS.md`를 보고 활성 작업과 계획 파일 위치를 찾는다. 세부 진행은 각 계획 파일의 체크리스트에서 확인한다. 계획 상태가 바뀌면 `PLANS.md`와 세부 계획 파일을 함께 갱신한다.

---

## 다음 중 떠오른 것

작업이 여러 갈래로 늘어나면 루트 인덱스가 없을 때 Codex가 오래된 문서나 특정 handoff에 과하게 의존할 수 있다. `PLANS.md`는 프로젝트의 현재 작업판 역할을 하고, 개별 plan은 실제 실행 기준 역할을 하게 두는 편이 좋다.
