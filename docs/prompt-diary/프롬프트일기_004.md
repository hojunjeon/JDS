# [프롬프트 일기 #4] UI 개편 작업을 세션 인수인계 가능한 흐름으로 바꾸기

> JDS를 만들면서 AI와 어떻게 협업하는지 기록한다.
> 어떤 프롬프트가 들어왔고, 무엇이 바뀌었고, 그래서 어떤 결정을 남겼는지 적는다.

---

## 그 시점의 게임

JDS는 01-10번 HTML UI 시안을 만든 뒤, `the former UI Renewal roadmap`를 기준으로 실제 production UI 개편을 시작한 상태였다. 첫 TDD 슬라이스로 `src/ui/theme.ts`를 만들고, `MenuScene`을 새 UI 토큰 기반으로 바꾸었다.

---

## 그날 상황

사용자는 세션이 초기화되어도 다음 Codex가 같은 계획과 검증 기준으로 이어갈 수 있기를 원했다. 기존 계획 문서에는 전체 방향은 있었지만, “마지막으로 무엇이 완료됐는지”, “다음에 어떤 테스트부터 쓰면 되는지”, “어떤 파일을 먼저 읽어야 하는지”가 충분히 선명하지 않았다.

---

## 입력된 프롬프트

> "세션이 초기화 되어도 계획에 따라 진행할 수 있게 인수인계 가능한 상태로 수정해줘."

---

## 무엇이 변화했나

UI 개편 작업은 단순히 화면을 예쁘게 바꾸는 작업이 아니라, 세션이 바뀌어도 계속 이어갈 수 있는 장기 워크플로우가 되었다.

그래서 `the former UI Renewal roadmap` 상단에 `Session Handoff` 섹션을 추가했고, 별도의 `the former UI Renewal handoff`를 만들었다. 이 문서에는 읽어야 할 파일 순서, 현재 완료 상태, 마지막 검증 결과, 다음 TDD 슬라이스, 중단 조건을 적었다.

---

## 그래서 어떤 결정이 남았나

다음 세션은 바로 구현을 추측하지 않는다. 먼저 `AGENTS.md`, 관련 JDS skills, `DESIGN.md`, `the former UI Renewal roadmap`, `the former UI Renewal handoff`를 읽고 이어간다.

다음 production 작업은 Phase 2가 아니라 Phase 1의 메뉴 흐름 모델이다. `src/ui/menuFlow.ts`와 `tests/menuFlow.test.ts`를 만들고, `start`, `stage-select`, `weapon-select` 상태 전환을 TDD로 잡은 뒤 `MenuScene`에 연결한다.

---

## 다음에 떠올릴 것

긴 UI 개편은 결과물보다 “다음 사람이 틀리지 않게 이어갈 수 있는 상태”가 중요하다. 계획 문서에는 목표뿐 아니라 마지막 검증, 다음 테스트, 멈춰야 할 조건까지 있어야 한다.
