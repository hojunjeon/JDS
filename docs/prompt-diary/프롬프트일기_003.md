# [프롬프트 일기 #3] 서브에이전트 운영 기준을 정하기로 한 날

> 이 시리즈는 JDS를 만들면서 AI와 어떻게 협업하는지를 기록한다.
> 어떤 프롬프트가 들어왔고, 무엇이 변화했고, 그래서 어떤 결정을 남겼는지 적는다.

---

## 그 시점의 게임

*JDS에는 이미 `AGENTS.md`, project-local skills, `DESIGN.md`, prompt diary가 있었다. 하지만 작업 흐름은 기본적으로 하나의 Codex 메인 에이전트가 전체 과정을 통괄하는 방식이었다.*

---

## 그날 상황

사용자는 기존 `superpowers` 중심 워크플로우에 서브에이전트를 도입하면 어떤 영향이 있을지 물었다. 그 다음에는 Explorer, Design Reviewer, Worker, Verifier 네 가지 시작 역할을 실제 운영 기준으로 만들고 싶다고 했다.

---

## 입력된 프롬프트

> *"4가지 서브에이전트를 생성하고 작업 크기별 서브에이전트 호출 기준을 정해줘."*

---

## 뭐가 나왔나

서브에이전트는 JDS의 기본 작업 방식을 대체하지 않고, 메인 에이전트가 필요할 때 호출하는 보조 역할로 정의했다.

- `Explorer`: 읽기 전용 프로젝트 조사
- `Design Reviewer`: `DESIGN.md`, UI/UX, 게임플레이 기준 검토
- `Worker`: 명확히 제한된 파일 또는 모듈 구현
- `Verifier`: 구현 후 독립 검증

또한 작업 크기를 Small, Medium, Large로 나누고, 크기와 위험도에 따라 어떤 서브에이전트를 쓸지 정했다.

---

## 그래서 어떤 결정을 했나

`AGENTS.md`에는 서브에이전트를 선택적으로 사용하라는 핵심 규칙만 추가했다. 세부 역할, 프롬프트 형태, 작업 크기별 호출 기준은 `docs/subagents.md`에 분리했다. 실제 Codex 작업 흐름에 영향을 주도록 `.codex/skills/jds-development-workflow/SKILL.md`에도 요약 기준을 추가했다.

앞으로 JDS에서는 작은 작업은 메인 에이전트 단독으로 처리하고, 중간 이상 작업부터 Explorer와 Verifier를 우선 고려한다. UI, HUD, 시각 효과, 게임플레이 변경에는 Design Reviewer를 추가한다. Worker는 파일 소유권이 명확할 때만 쓴다.

---

## 다음에 떠올린 것

서브에이전트가 많아지면 작업이 더 안전해질 수 있지만, 작은 일까지 모두 분산하면 느려진다. 그래서 JDS에는 "항상 병렬화"가 아니라 "작업 크기와 위험도에 맞춘 선택적 병렬화"가 더 어울린다.
